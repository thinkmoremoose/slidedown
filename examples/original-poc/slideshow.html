instead of learning to drive PowerPoint

The Current Approach

How we teach AI to make presentations today:

  • 🔧 Learn PowerPoint's COM API and OOXML format
  • 🔧 Learn Google Slides' REST API and OAuth flow
  • 🔧 Learn Keynote's AppleScript interface
  • 🔧 Then Canva, Prezi, Figma, Pitch…

Each framework is a new language to master.
Each update is a new way to break.

The current paradigm is teaching AI to operate existing tools — essentially building AI-powered wrappers around legacy software. Every time a framework updates, every integration risks breaking. We're on a treadmill, and the treadmill keeps accelerating.

The Abstraction Tax

Every framework demands AI learn:

  • 📦 A proprietary file format (OOXML, Protobuf, JSON blobs…)
  • 🔌 A unique API surface — hundreds of endpoints
  • 🧩 A plugin and extension ecosystem
  • 🔄 Version-specific quirks and breaking changes
  • 📚 Thousands of pages of documentation — per tool

More effort adapting to the tool than creating value.

PowerPoint alone has over 300 distinct object types in its API. Google Slides has a completely different object model. Keynote barely has an API at all. Every framework is a walled garden with its own rules, and AI has to learn them all — separately.

What Do We Actually Need?

Strip it down to first principles. A presentation needs to:

  • ✓ Display content on a screen
  • ✓ Navigate between pages
  • ✓ Support speaker notes
  • ✓ Export to PDF
  • ✓ Work offline

That's it. That's the whole requirements list.

We've conflated "presentation software" with "the ability to present." PowerPoint does a thousand things. We need five. Why force AI to learn a thousand things when we only need five?

What if AI just…
built the app?

HTML → content & structure

CSS → styling & animation

JavaScript → interaction & logic

Browser → the universal runtime

One file. Zero dependencies. Infinite flexibility.

The browser is the most widely deployed application runtime in human history. Every phone, tablet, laptop, and desktop has one. HTML has been backward-compatible for 30 years. Instead of wrapping a framework, just generate the output directly.

Zero Dependencies

  • 🚫 No install, no license, no subscription
  • 💻 Works on every OS — if it has a browser, it works
  • ✈️ Works offline — it's just a file on your disk
  • 🏛️ Will still work in 20 years — HTML is eternal
  • 📦 This entire presentation: < 30 KB

Compare: PowerPoint requires a 4 GB install and $160/year.

The durability point is key. HTML files from 1995 still render correctly in modern browsers. Can you open a PowerPoint 95 file today? Maybe — with compatibility warnings, broken fonts, and missing features. HTML just works. Forever.

Total Control

❌ Framework-driven

  • "Sorry, the template doesn't support that layout"
  • "Animations are limited to these 12 presets"
  • "Custom fonts require a plugin"
  • "You can't nest that object there"

✓ Purpose-built

  • Every pixel is intentional
  • Every animation is custom
  • Every interaction is designed for this specific need
  • The only limit is the web platform
Anyone who's fought PowerPoint's auto-layout or tried to pixel-perfect-align objects knows this pain. When AI builds the app, it designs for YOUR content, not for a generic template's assumptions.

The Web Platform IS the Framework

Need something? Just add it.

  • 📊 Need a chart? → <canvas> or <svg>
  • ❓ Need a quiz? → 20 lines of JavaScript
  • 🎮 Need 3D? → WebGL, right in the browser
  • 📡 Need live data? → fetch() API
  • ⏱️ Need a timer? → setInterval()
  • 🎵 Need audio? → <audio> element

Try embedding a live API call in a PowerPoint slide.

The browser is the most capable application platform ever built. Every year it gets more powerful. PowerPoint hasn't fundamentally changed in two decades. When you build directly for the browser, you get access to the entire modern web platform — for free.

But What About…?

  • "Collaboration?" → Git + any text editor. Better version control than Google Slides.
  • "Non-technical users?" → AI generates the file for them. That's the whole point.
  • "Brand templates?" → CSS custom properties. Swap one variable block.
  • "Editing in place?" → That's just another app to build.
  • "Existing content?" → AI can convert formats too.

None of these are blockers. They're all solvable — and most are already easier than the PowerPoint equivalents.

The collaboration objection is the most common, but think about it: Google Docs solved collaboration not by teaching AI to use Word's COM API, but by building a new purpose-built app. The same logic applies here.
